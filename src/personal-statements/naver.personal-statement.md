1. 지원 포지션에서 수행할 업무를 고려하여, 최우선적으로 해결하고 싶은 과제
네이버 Data Engineer 포지션에서 가장 먼저 풀고 싶은 과제는, 대규모 서비스 로그를 신뢰할 수 있는 지표와 데이터 제품으로 연결하는 기준을 단단히 세우는 일입니다. 서비스가 커질수록 파이프라인을 하나 더 만드는 일보다 중요한 것은 어떤 이벤트를 남길지, 어떤 모델로 가공할지, 같은 지표를 여러 팀이 같은 뜻으로 보게 만들지를 먼저 맞추는 일이라고 생각합니다. 수집은 잘되는데 해석이 제각각이면 데이터는 많아도 의사결정은 느려지고, 팀 사이 논의도 반복됩니다.

그래서 저는 Spark·Airflow 같은 처리 기술 자체보다도, 로그 설계, 데이터 모델링, 품질 검증, 공통 지표 정의가 한 흐름으로 이어지는 구조를 우선 만들고 싶습니다. 특히 서비스팀, 분석팀, 플랫폼팀이 각자 다른 언어로 말하지 않도록 지표 정의와 데이터 책임 경계를 분명히 하는 일이 중요하다고 봅니다. 거래소 환경에서 CDC·스트리밍 파이프라인, 지표 저장소, self-serve 데이터 환경을 운영하며 느낀 것도 결국 같은 점이었습니다. 데이터를 빨리 쌓는 것보다, 필요한 사람이 믿고 바로 쓸 수 있게 만드는 일이 더 어렵고 더 중요했습니다. 네이버에서는 더 큰 트래픽과 더 다양한 서비스 환경에서 이 기준을 다듬고, 데이터가 쌓이는 속도만이 아니라 실제 활용되는 품질까지 함께 끌어올리는 역할을 맡고 싶습니다. 그래야 데이터가 보고용 자산에 머물지 않고, 서비스 개선과 의사결정에 바로 쓰이는 기반이 된다고 생각합니다.

2. AI를 활용하여 문제를 해결했거나, 해결을 시도해 본 사례
AI를 가장 실무적으로 써 본 경험은, 새로운 데이터 플랫폼 구조를 검토할 때 Agent를 리서치 보조 도구로 붙여 본 일입니다. 예를 들어 Reverse ETL과 별도 서빙 DB 중 심 구조가 운영 복잡도를 키운다고 느꼈을 때, ClickHouse·Druid·Pinot 같은 OLAP 엔진을 대안으로 검토했습니다. 이때 Agent에게 각 오픈소스의 코드 구조, 핵심 모듈 역할, 운영 시 주의할 점을 먼저 정리하게 하고, 저는 그 결과를 공식 문서와 저장소를 다시 보며 걸러냈습니다. 덕분에 반복적인 탐색에 쓰는 시간을 줄이고, 우리 환경 에서 품질 책임을 어디에 둘지, 장애가 났을 때 복구가 쉬운지, 관측성을 어떻게 확보할지 같은 판단에 더 오래 집중할 수 있었습니다.

특히 기능 비교표만 보는 수준에 서 끝내지 않고, 실제 운영 시 병목이 생길 지점과 팀이 감당해야 할 복잡도를 먼저 따져 볼 수 있었습니다. 이런 방식은 새로운 도구를 빠르게 훑고 버리는 데서 끝나지 않고, 우리 환경에 맞는 선택지를 좁히는 데 도움이 됐습니다. 실제로 검토 초반의 방향을 잡는 속도도 눈에 띄게 빨라졌습니다. AI가 결론을 대신 내려준 것은 아니지 만, 비교와 정리 속도를 높여서 아키텍처 검토를 더 깊고 현실적으로 밀어붙이는 데 분명한 도움이 됐습니다. 지금도 저는 AI를 답변 생성기보다, 판단 전에 선택지를 빠 르게 좁히고 검토의 밀도를 높여 주는 도구에 가깝게 쓰고 있습니다.
