# 현대자동차 지원 문항 답변

## 1. 대규모 데이터 플랫폼 또는 배치/스트리밍 파이프라인을 설계·고도화했던 경험 중 가장 임팩트가 컸던 사례를 설명하고 당시 해결하려는 문제, 본인이 내린 핵심 아키텍처 의사결정, 품질·비용·개발속도 간 트레이드오프를 어떻게 판단했는지 기술해 주세요.

Bithumb에서 Lakehouse 기반 데이터 플랫폼을 고도화하며 배치 중심 구조를 Kafka CDC, Flink, Iceberg 기반 준실시간 처리 구조로 확장한 경험이 가장 임팩트가 컸습니다. 기존에는 Airflow 배치와 DW 집계로 내부 분석 데이터를 제공했지만, 서비스와 마케팅 영역에서 더 짧은 주기의 데이터와 비즈니스 로직이 반영된 가공 데이터 수요가 늘었습니다. Kafka Sink Connector만으로는 변경 이력, 최신 상태 테이블, 복잡한 로직을 함께 다루기 어려웠습니다.

핵심 결정은 세 가지였습니다. CDC 이벤트는 Flink Table API로 처리해 Iceberg에 적재했고, changelog와 materialized 적재 모드를 나눠 원본 이력과 최신 상태 테이블의 책임을 분리했습니다. 운영은 Managed Flink로 시작하되 애플리케이션 구조, 설정, 테이블 naming rule, 배포 단위는 코드로 관리했습니다. 품질은 offset, schema, 재처리, 정합성을 우선했고 비용은 직접 운영보다 높더라도 초기 클러스터 운영 리스크와 학습 비용을 줄이는 쪽을 택했습니다. 개발속도는 Table API와 multi-module boilerplate로 보완했습니다. 이후 EKS, Helm, ArgoCD와 SAD/ADR까지 정리해 같은 기준으로 확장 가능한 플랫폼 패턴을 만들었습니다.

## 2. 비즈니스/운영 문제를 데이터 제품, 지표 체계, 또는 분석 기반 프로세스로 해결한 경험을 설명하고, 협업한 이해관계자, 본인의 역할, 결과적으로 개선된 지표 또는 운영 효과를 서술해 주세요.

Coinone에서 지표 저장소와 self-serve data 환경을 구축해 반복적인 데이터 요청과 지표 해석 차이를 줄인 경험이 있습니다. 당시 서비스/마케팅, 보안, 운영 조직은 필요한 데이터를 얻을 때마다 데이터팀에 요청해야 했고, 지표 정의도 사람이나 문서에 흩어져 있었습니다. 데이터 위치와 메타데이터를 찾기 어려워 분석 리드타임이 길었고, 외부 기관 보고용 정제 작업처럼 반복되는 업무도 수동 처리 비중이 높았습니다.

저는 데이터 엔지니어로서 합의된 지표를 CLI로 조회하는 데이터 제품과 지표 저장소 구조를 설계했습니다. DataHub, Airbyte, MageAI를 Kubernetes 환경에 배포해 데이터 탐색, 외부 데이터 수집, workflow 자동화를 self-serve 흐름으로 연결했습니다. DataHub에는 RDB information schema, dbt, BigQuery 메타데이터를 수집했고 보안팀이 개인정보 tag를 직접 관리하도록 했습니다. Airbyte custom connector로 외부 거래소 candle 데이터 수집을 표준화했고, MageAI로 보고용 정제 작업을 자동화했습니다. 결과적으로 데이터팀은 요청 처리보다 플랫폼 제공에 집중할 수 있었고, 현업은 필요한 데이터와 지표를 더 빠르게 찾고 재사용할 수 있게 되었습니다.
