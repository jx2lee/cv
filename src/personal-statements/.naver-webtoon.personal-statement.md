# 네이버웹툰 지원서 문항 답변

## 1. 본인의 커리어 흐름에서 지금 이 포지션을 선택한 이유를 적어주세요. 지금까지의 경험과 앞으로 하고 싶은 일이 이 포지션과 어떻게 연결되는지 알려주시면 좋습니다.

제 커리어는 데이터를 다루는 일에서 출발해, 데이터를 안정적으로 흐르게 만드는 일로 자연스럽게 이어져 왔습니다. 초기에 TmaxData에서는 DBMS와 빅데이터 제품 기술 지원을 담당하며 데이터 문제를 구조적으로 진단하는 습관을 익혔고, NHN Enterprise에서는 클라우드 환경에서 서비스 운영과 배포 체계를 경험하며 프로덕션 안정성의 중요성을 배웠습니다. 이후 Coinone과 Bithumb에서는 데이터 엔지니어로서 데이터웨어하우스, CDC, 이벤트 스트리밍, 데이터 품질 관리, 셀프서브 데이터 환경, 데이터 API 서빙까지 데이터가 수집되고 가공되어 실제 서비스와 분석에 활용되기까지의 흐름을 직접 설계하고 운영해 왔습니다.

지금 이 포지션을 선택한 이유는, 그동안 쌓아 온 경험이 네이버웹툰과 같은 대규모 서비스 환경의 데이터 플랫폼 역할과 맞닿아 있다고 판단했기 때문입니다. 저는 배치와 스트리밍 파이프라인을 만드는 데 그치지 않고, 운영 과정에서 발생하는 지연, 오류, 복구, 품질 저하 문제를 꾸준히 해결해 왔습니다. Coinone에서는 CDC와 이벤트 파이프라인을 운영하며 적재 성공률 99.9%를 유지할 수 있는 관측 체계와 복구 기준을 마련했고, Bithumb에서는 데이터 품질 게이트를 도입해 파이프라인 실패율을 낮추고 데이터 API 구조를 설계하며 데이터 결과물이 실제 사용자 경험으로 이어지는 지점까지 다뤘습니다.

앞으로는 더 큰 규모의 사용자 트래픽과 다양한 데이터 흐름이 존재하는 환경에서, 단순히 파이프라인을 구축하는 엔지니어를 넘어 서비스팀과 분석조직이 믿고 사용할 수 있는 데이터 플랫폼의 기준을 만드는 역할을 하고 싶습니다. 네이버웹툰은 글로벌 사용자와 다양한 콘텐츠 소비 흐름이 만들어 내는 방대한 데이터를 다루는 서비스이고, 데이터가 실제 서비스 개선과 의사결정에 연결되는 속도와 품질이 중요하다고 생각합니다. 지금까지의 경험이 운영 가능한 데이터 흐름을 만드는 데 있었다면, 앞으로는 네이버웹툰에서 그 흐름을 더 큰 규모와 더 높은 신뢰 수준으로 확장하고 싶습니다.

## 2. 지금까지 진행한 프로젝트 중 본인이 주도해서 풀어낸 문제 하나를 자유롭게 소개해 주세요. 어떤 상황이었고, 어떻게 접근했고, 결과적으로 무엇이 달라졌는지를 중심으로 적어주시면 됩니다. (기술 스택보다 문제 해결 과정이 더 궁금합니다)

가장 인상 깊었던 문제 해결 경험은 Coinone에서 Debezium 기반 CDC 파이프라인을 구축한 일입니다. 당시 분석팀과 서비스팀은 변경 이력이 살아 있는 데이터를 원했지만, 기존 DW 적재 방식으로는 삭제나 갱신 이벤트까지 포함한 실제 변경 흐름을 확보하기 어려웠습니다. 로그 테이블도 있었지만 비즈니스 로직과 어긋나는 경우가 있어 신뢰하기 어려웠습니다. 결국 문제는 단순히 CDC를 도입하는 것이 아니라, 운영 중인 RDS에 영향을 주지 않으면서 이 파이프라인을 지속적으로 운영 가능한 형태로 만드는 것이었습니다.

저는 이 문제를 세 가지로 나눠 접근했습니다. 첫째, 정확한 데이터를 가져오는 것. 둘째, 서비스에 부담을 주지 않는 것. 셋째, 장애가 발생해도 빠르게 원인을 찾고 복구할 수 있게 만드는 것이었습니다. 먼저 DBA와 함께 snapshot과 lock 정책을 조정해 운영 DB 영향도를 낮췄고, 커넥터 설정과 배포 방식을 표준화해 변경 이력을 코드처럼 관리할 수 있게 했습니다. 이후에는 단순히 파이프라인이 동작하는 수준에 머무르지 않고, 실제로 무엇이 얼마나 지연되고 있는지 확인할 수 있도록 관측 지표와 알림 체계를 구축했습니다. 마지막으로 binlog 포지션 변경이나 재시작 같은 운영 이슈가 생겼을 때 담당자가 같은 방식으로 대응할 수 있도록 복구 절차와 문서를 정리했습니다.

그 결과 한 달 기준 99.9% 적재 성공률을 유지했고, snapshot이나 lock으로 인한 DB 성능 이슈도 발생하지 않았습니다. 더 중요한 변화는 팀이 CDC를 불안한 신규 기술이 아니라 반복해서 확장할 수 있는 표준 파이프라인으로 보기 시작했다는 점입니다. 이 경험을 통해 문제 해결은 기술 선택만으로 끝나지 않으며, 운영 가능성, 관측 가능성, 복구 가능성까지 함께 설계해야 조직에 남는 결과를 만들 수 있다는 점을 배웠습니다.

## 3. 다른 직군/팀과 함께 일하며 갈등이나 어려움이 있었던 경험과, 그 상황에서 본인이 어떻게 행동했는지 알려주세요. 잘 풀린 사례뿐 아니라 잘 안 풀렸던 사례도 좋습니다. 무엇을 배웠는지가 더 중요합니다.

Bithumb에서 데이터 API를 만들던 과정에서 서비스팀과 협업하며 어려움을 겪은 경험이 있습니다. 서비스팀은 사용자별 집계 데이터를 가능한 한 빠르게 조회할 수 있기를 원했고, 처음에는 필요한 값이 계속 늘어나는 방향으로 요구가 들어왔습니다. 반면 저는 DW 집계 결과를 그대로 서비스에 연결하면 운영 복잡도와 장애 위험이 커질 수 있다고 판단했습니다. 같은 목표를 보고 있었지만, 서비스팀은 더 유연한 제공 방식을 원했고 저는 운영 가능한 구조를 우선시하면서 시각 차이가 생겼습니다.

초기에는 저 역시 기술적으로 왜 어려운지를 설명하는 데 집중했습니다. 하지만 그렇게 접근하니 상대 입장에서는 안 되는 이유를 먼저 말하는 사람처럼 보였고, 대화도 빠르게 진전되지 않았습니다. 이후에는 접근 방식을 바꿨습니다. 어떤 데이터를 언제까지, 어느 수준의 정확도로 제공해야 실제 사용자 경험에 도움이 되는지부터 다시 정리했고, 서비스팀과 함께 SLA와 SLO를 먼저 맞췄습니다. 그 위에서 무거운 집계는 DW에서 처리하고, 서비스에는 조회에 필요한 최소 형태만 전달하는 구조로 책임 경계를 분리했습니다. 또한 운영 중 어떤 상황에서 누가 확인하고 대응할지까지 문서와 런북으로 남겨, 논의가 추상적인 선호가 아니라 운영 기준 위에서 이뤄지도록 만들었습니다.

그 결과 서비스팀이 원하는 속도와 데이터팀이 지켜야 할 안정성 사이에서 현실적인 균형점을 찾을 수 있었고, 실제로 DW 집계 SLA 99.9% 수준의 구조를 운영할 수 있었습니다. 이 경험을 통해 협업 갈등은 기술적 정답이 부족해서라기보다 서로의 성공 기준이 다를 때 생긴다는 점을 배웠습니다. 그래서 지금은 다른 직군과 일할 때 제 해결책을 먼저 설명하기보다, 상대가 중요하게 보는 사용자 경험과 운영 리스크를 먼저 맞춘 뒤 구조를 제안하려고 합니다.
