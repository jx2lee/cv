\ProjectTitle{데이터 플랫폼 on kubernetes}{2026.04 - }

\textbf{Description}: 데이터 플랫폼 운영 환경을 Kubernetes|EKS 환경으로 옮기는 작업을 주도했다.
플랫폼 컴포넌트의 배포 구성을 Helm Chart 로 관리하고, ArgoCD 기반 GitOps 패턴을 적용할 수 있도록 배포 구조를 정리했다.

\resumeSubProjectListStart
  % --- Problem ---
  \resumeSubProject{\textbf{Problem}}{%
    \resumeSubBulletsStart
      \item ASG(EC2) 기반의 서비스를 제공하며 리소스 스케일아웃 에 대한 관리 복잡도가 증가함
      \item Managed 서비스 이용 및 비 컨테이너 환경으로 구성된 플랫폼으로 환경별 차이를 추적하기 어렵고 데이터팀 CICD 작업의 비효율이 증가함
      \item 플랫폼 운영 컴포넌트가 늘어날수록 배포 방식, Secret 관리, 환경별 설정을 같은 기준으로 유지하기 어려움(코드로 관리할 수 없음)

      \item ArgoCD, AWS Load Balancer Controller, Secrets Store CSI Driver 등 플랫폼 운영 컴포넌트의 Chart 및 환경별 values 정리
      \item Observability|Datadog(datadog agent operator)

  }

      \item Helm values 와 ArgoCD 매니페스트를 기준으로 환경별 설정 차이, 배포 구성, Secret 참조 방식을 코드에서 함께 확인할 수 있게 정리
      \item 신규 플랫폼 컴포넌트 추가 시 공통 Chart 구조와 GitOps 배포 흐름을 재사용할 수 있도록 운영 기준을 표준화
    \resumeSubBulletsEnd
  }

      \item 환경별 구성 차이와 Secret 관리 기준이 명확해져 데이터팀의 배포 검토 및 CICD 운영 부담을 낮춤
      \item 이후 Flink, Kafka Connect 등 데이터 플랫폼 컴포넌트에도 같은 배포/운영 패턴을 적용할 수 있는 기반 확보
    \resumeSubBulletsEnd
  }

  % --- Self-reflection ---
  \resumeSubProject{\textbf{Self-Reflection}}{%
    \resumeSubBulletsStart
      \item 애플리케이션을 이미지로 만드는 일에 그치지 않고, 배포 방식과 운영 설정 관리까지 함께 바꾸는 작업임을 체감
      \item 플랫폼 컴포넌트|Helm Chart, ArgoCD, Secret 관리가 맞물려야 안정적인 플랫폼 운영이 가능하다는 점을 배움
      \item 데이터 엔지니어링 업무에서도 Kubernetes 이해가 중요해지고 있음을 실무적으로 경험
    \resumeSubBulletsEnd
  }

  \resumeSubProject{\textbf{Architecture Decision}}{%
    \resumeSubBulletsStart
      \item \textbf{Kubernetes}: ASG 에 사용한 스펙을 관리하지 않고 선언형 방식의 오케스트레이션이 필요했음.
        작업 변경|특히 resource 이 코드로 관리할 수 있었음.
        오버엔지니어링으로 느낄 수 있지만 기존 SaaS 서비스를 내제화 하여 플랫폼 컴포넌트에 대한 운영 경험을 쌓을 수 있음(Not Blackbox)
      \item \textbf{Helm/ArgoCD}: 플랫폼 컴포넌트가 늘어날수록 manifest 를 직접 복사해 관리하는 방식으로는 환경별 설정 차이와 변경 이력을 일관되게 추적하기 어렵다고 판단함.
        Helm 으로 공통 리소스 구조와 환경별 values 를 분리하고, ArgoCD 로 Git 을 기준으로 한 선언형 배포 흐름을 구성함.
        별도 배포 도구를 직접 만들기보다 Kubernetes 생태계에서 널리 쓰이는 조합(de facto)을 채택해 운영 부담과 팀 내 학습 비용을 줄이는 방향을 선택함.
    \resumeSubBulletsEnd
  }

\ProjectTitle{Flink 기반 CDC Iceberg Sink 개발}{2026.04 - 2026.05}

\textbf{Description}: Airflow orchestration 코드와 분리해 Managed Flink 애플리케이션을 관리할 수 있는 구조를 만들었다.
Kafka CDC 이벤트를 Flink Table API로 읽어 Iceberg 테이블에 적재하는 Sink 애플리케이션|changelog/materialized 을 개발했다.
매니지드 서비스를 사용할 수 밖에 없는 환경에 리소스를 코드로|Terraform 정리했고 애플리케이션 코드 변경으로 발생하는 CICD 파이프라인도 마련했다.

\resumeSubProjectListStart
  % --- Problem ---
  \resumeSubProject{\textbf{Problem}}{%
    \resumeSubBulletsStart
      \item 시간단위 배치보다 더 짧은 주기|Interval 의 데이터 제공의 수요가 증가함
      \item Kafka Sink Connector 로 간단한 데이터 파이프라인을 제공할 수 있었지만 비즈니스로직을 담은 수요도 함께 증가
    \resumeSubBulletsEnd
  }

  % --- Action ---
  \resumeSubProject{\textbf{Action}}{%
    \resumeSubBulletsStart
      \item multi-module 기반 Flink 애플리케이션 구성 및 JAR 빌드 환경
      \item Table API 기반|Everything is SQL 구현: CDC 메세지를 읽어 Iceberg 테이블로 적재
      \item changelog/materialized 적재 모드 지원으로 원본 이벤트 누적과 최신 상태 테이블 구성을 분리
      \item 복잡한 비즈니스 로직을 담을 수 있는 애플리케이션 boilerplate 구성
    \resumeSubBulletsEnd
  }

  % --- Result ---
  \resumeSubProject{\textbf{Result}}{%
    \resumeSubBulletsStart
      \item Kafka CDC 이벤트를 Flink Table API 로 처리해 Iceberg 테이블에 적재하는 재사용 가능한 Sink 애플리케이션을 구현
      \item changelog 와 materialized 적재 모드를 분리해 원본 이벤트 보관과 최신 상태 테이블 구성을 같은 애플리케이션 구조에서 지원
      \item multi-module 구조와 JAR 빌드 환경을 정리해 Airflow DAG 와 분리된 Flink 애플리케이션 개발/배포 단위를 마련
      \item Managed Flink 실행 설정과 테이블 naming rule 을 코드와 설정으로 관리해 환경별 운영 차이를 줄임
    \resumeSubBulletsEnd
  }

  % --- Impact ---
  \resumeSubProject{\textbf{Impact}}{%
    \resumeSubBulletsStart
      \item 시간 단위 배치로 제공하던 데이터를 더 짧은 주기로 적재할 수 있는 스트리밍 기반 처리 경로를 확보|a.k.a 준실시간
      \item Kafka Sink Connector 만으로 처리하기 어려운 비즈니스 로직을 Flink 애플리케이션 레이어에서 다룰 수 있게 됨
      \item Kafka CDC, Flink, Iceberg 로 이어지는 Lakehouse 적재 패턴을 구축해 이후 실시간성 데이터 파이프라인 확장의 기준을 마련
      \item Airflow orchestration 중심의 운영 범위를 stream processing, table format, catalog 관리까지 확장
    \resumeSubBulletsEnd
  }

  % --- Self-reflection ---
  \resumeSubProject{\textbf{Self-Reflection}}{%
    \resumeSubBulletsStart
      \item Flink 애플리케이션은 DAG 스크립트와 달리 빌드 산출물, 런타임 설정, 테이블 스키마를 함께 고려해야 한다는 점을 체감
      \item CDC 이벤트를 Iceberg 테이블로 적재하며 append-only 로그와 materialized view 성격의 테이블 설계 차이를 더 명확히 이해
      \item 데이터 파이프라인 운영 범위가 orchestration 을 넘어 stream processing, table format, catalog 관리까지 확장된다는 점을 경험
    \resumeSubBulletsEnd
  }

  % --- Architecture Decision Recordsㅡsimple ---
  \resumeSubProject{\textbf{Architecture Decision}}{%
    \resumeSubBulletsStart
      \item \textbf{Apache Flink}: 단순 Sink 처리만으로는 CDC 이벤트에 필요한 비즈니스 로직을 충분히 담기 어렵다고 판단.
          ksqlDB 도 선택지로 볼 수 있었지만, 복잡한 변환 로직을 코드로 표현하고 테스트 가능한 애플리케이션 단위로 관리하기에는 Flink 가 더 적합하다고 판단.
          제이슨 크랩스에 따르면 더 이상 Confluent 핵심 리소스로 보지 않는다고 선언함.
          Table API 를 활용해 SQL 기반의 처리 흐름은 유지하면서도, 필요할 때 Java/Scala/Python 코드로 로직을 확장할 수 있는 구조를 선택함. (ksqlDB|Only SQL)
        \item \textbf{Managed Service}: 초기에는 Kubernetes Operator 기반 배포도 고려했지만, Flink 클러스터 운영까지 팀이 직접 감당하면 애플리케이션 개발보다 운영 복잡도가 먼저 커질 수 있다고 판단함.
          런타임 운영 부담을 줄이고 팀이 CDC 처리 로직과 Iceberg 적재 구조에 집중할 수 있도록 Managed Flink 를 선택함. (But|결국 내부구조를 알아야 한다는 것을 뼈저리게 학습)
          이후 운영 경험이 쌓이면 Kubernetes 기반 배포로 확장할 수 있도록 애플리케이션 구조와 설정 주입 방식을 분리함.
        \item \textbf{Catalog|Glue}: Iceberg 테이블을 S3 warehouse 와 함께 운영해야 했기 때문에 AWS 환경과 자연스럽게 연결되는 Glue Catalog 를 사용함.
          국내 빅테크 기업의 경우 자체 구현한 카탈로그를 사용하지만 도입 초기 단계에서 많이 사용되는 카탈로그로도 충분히 사용가능하다 판단함.
    \resumeSubBulletsEnd
  }

      \item CI `changes` 조건을 활용해 변경된 프로젝트만 선택적으로 빌드|테스트|배포하는 스크립트 구성

      \item 프로젝트마다 달랐던 개발 환경과 CI/CD 구성을 맞춰 신규 파이프라인 셋업 부담을 줄임
      \item 변경된 프로젝트만 빌드/테스트/배포하도록 구성해 Monorepo 전환 이후에도 CI/CD 실행 범위를 제어
      \item 공통 코드와 파이프라인 코드의 변경 영향도를 한곳에서 확인할 수 있어 리뷰와 배포 전 검토가 쉬워짐

  \resumeSubProject{\textbf{Problem}}{%
    \resumeSubBulletsStart
      \item 내부 여러 조직이 필요한 데이터를 얻기 위해 매번 데이터셀에 요청하면서 데이터 활용 과정에 병목이 발생
      \item RDB, DW, 외부 API 등 데이터 위치를 파악하기 어렵고 메타데이터 관리가 부족해 재사용성이 낮았음
      \item 외부 데이터(거래소 API, CMC 등)를 개별 구현 방식으로 수집하면서 신뢰성과 유지보수에 부담이 있었음
      \item 비전문가도 사용할 수 있는 직관적인 self-serve 데이터 환경이 부족했음
    \resumeSubBulletsEnd
  }

  % --- Action ---
  \resumeSubProject{\textbf{Action}}{%
    \resumeSubBulletsStart
      \item DataHub 구축:
        \resumeThirdListStart
          \item Airflow 스크립트로 RDB information schema ingestion 을 자동화
          \item dbt/BigQuery 는 recipe YAML 과 ingestion CLI 기반으로 메타데이터를 수집
          \item Helm/ArgoCD 기반 배포와 RBAC, tag 정책을 구성
        \resumeThirdListEnd
      \item Airbyte 도입:
        \resumeThirdListStart
          \item Python CDK 기반 custom connector 를 개발하고 connector 저장소를 구성
          \item semantic versioning 을 도입하고 GitHub Actions 에서 ECR 로 자동 빌드/배포
          \item OAuth 기능 부재를 보완하기 위해 oauth2-proxy 를 연동
        \resumeThirdListEnd
      \item MageAI 도입:
        \resumeThirdListStart
          \item 외부 기관 보고용 정제 작업과 workflow 자동화를 구성
          \item 공식 이미지의 PostgreSQL client 누락 문제를 확인하고 직접 PR 을 제출해 merge 경험 확보
          \item Helm chart 를 수정해 shared-volume 기반 운영 구조를 구성
        \resumeThirdListEnd
      \item 데이터 적재(EL), 탐색(Browse), 메타데이터(Document), 변환(Transform)을 연결하는 Self-Serve 구조를 정리
    \resumeSubBulletsEnd
  }

  % --- Result ---
  \resumeSubProject{\textbf{Result}}{%
    \resumeSubBulletsStart
      \item 보안팀이 DataHub tag 기능으로 개인정보 레이블을 직접 관리할 수 있게 되어 데이터셀 요청을 줄임
      \item 외부 거래소 candle 데이터 수집을 Airbyte 기반으로 자동화해 수집 방식과 데이터 일관성을 정리
      \item MageAI 기반 workflow 로 외부 기관 보고용 정제 작업을 자동화하고 반복 수동 작업을 줄임
      \item 내부 사용자가 데이터 위치와 메타데이터를 직접 확인할 수 있는 self-serve 데이터 활용 기반을 마련
    \resumeSubBulletsEnd
  }

  % --- Impact ---
  \resumeSubProject{\textbf{Impact}}{%
    \resumeSubBulletsStart
      \item 데이터 조직의 역할을 요청 처리 중심에서 플랫폼 제공 중심으로 전환할 수 있는 기반을 마련
      \item DataHub, Airbyte, MageAI 등 OSS 를 Kubernetes 환경에서 운영하며 데이터 플랫폼 엔지니어링 역량을 강화
      \item 개별 조직이 필요한 데이터를 직접 찾고 활용할 수 있는 구조를 제공해 팀 간 데이터 활용 범위를 넓힘
      \item Helm/ArgoCD 기반 배포 방식을 표준화해 이후 신규 데이터 제품 도입 시 운영 일관성을 확보
    \resumeSubBulletsEnd
  }

  % --- Self-reflection ---
  \resumeSubProject{\textbf{Self-Reflection}}{%
    \resumeSubBulletsStart
      \item Self-Serve 플랫폼은 도구를 설치하는 작업이 아니라 사용자의 데이터 workflow 를 쉽게 만드는 일임을 이해
      \item 성숙한 OSS 와 초기 단계 도구를 함께 운영하면서 각 도구의 내부 구조와 제약을 깊게 분석할 수 있었음
      \item OAuth, Helm chart 수정, custom connector 개발 같은 운영 이슈를 해결하며 플랫폼 운영 역량을 넓힘
    \resumeSubBulletsEnd
  }
\resumeSubProjectListEnd
